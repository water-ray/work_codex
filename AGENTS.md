# Codex 项目规则



## 基本原则

- 默认使用中文沟通、命名说明和文档内容；代码标识符按对应语言和项目惯例处理。
- 先读取并验证上下文，如有冲突或者疑问点，先询问我再修改；不要基于未确认信息扩展需求。
- 只处理用户明确要求的范围；如果发现必要的关联清理，和本次变更一起完成。
- 引用项目内容时使用真实文件路径；除非用户要求，不展开无关实现细节。
- 复杂功能必须拆分多代码文件，禁止单文件编写各种不同的功能实现。

## 修改边界

- 遵循项目现有风格、目录结构和命名方式，有必要的重构直接询问我，并给出优化方案选择。
- 移除或替换功能、配置时，同步删除相关代码、类型、API、文档、测试、持久化字段、任务入口和用户可见字符串。
- 重构/修改框架时，不保留旧流程、死代码、弃用字段、兼容分支、重复别名、历史 fallback 或注释掉的代码，除非用户明确要求兼容。
- 如果必须保留兼容层，在同次变更中写清兼容范围、清理触发条件和移除计划。

## 目录约定

- 通用辅助目录统一使用 `scripts/`、`docs/`、`Bin/`、`temp/`。
- 项目脚本默认使用 Python，放入 `scripts/<用途>/`；不要散落在仓库根目录或源码目录。
- 项目说明、设计、使用文档放入 `docs/<项目或主题>/`；派生项目的根目录 `README.md` 可按项目需要自由组织，但不能替代 `AGENTS.md` 或 skill 中的开发规范。
- 编译、构建和打包后的最终产物输出到 `Bin/<项目>/`。
- 临时文件、下载、截图、UI dump、临时日志和人工测试输出写入仓库内 `temp/`，不要放到根目录或源码目录。
- 新增脚本或文档时优先复用现有子目录；只有确有新类别时再新增一级目录。

## Codex Skills

- 本仓库的项目内 skill 源位于 `.codex/skills/`。
- 如果当前 Codex 环境未自动列出项目内 skills，仍按文件路径读取本仓库的 `SKILL.md`；需要全局触发时，再把 `.codex/skills/<skill-name>/` 同步到 `$CODEX_HOME/skills/`。
- Skill 文档用于限制可重复执行的开发规范，包括项目/功能命名、目录布局、修改边界、验证方式、任务入口和失败处理。
- 新增项目或子项目时，必须创建或更新一个项目 skill。
- 新增、重构或实质修改稳定功能、复杂工作流、长期运行任务、跨模块接口或专用验证流程时，必须在同次变更中创建或更新对应功能 skill 或任务 skill；这是完成条件，不是可选后续。
- 修改已有功能且已有对应 skill 时，必须先读取并按该 skill 执行，完成后同步更新 skill 中的入口、规则、验证方式和失败处理。
- 如果没有对应 skill，应按 `codex-skill-generator` 规则新增；仅一次性临时实验可不写入 skill，但需要能说明为何不沉淀。
- Skill 名称使用小写字母、数字和连字符；项目 skill 使用 `<project-slug>-project`，功能 skill 使用 `<project-slug>-<feature-slug>`，避免空格、下划线、中文名和重复别名。

## 执行与验证

- 代码修改完成后，优先使用静态校验、语法检查、lint、类型检查或最小必要测试验证。
- 除非用户明确要求或验证确有必要，不自动启动完整服务、执行完整构建或生成大体积产物。
